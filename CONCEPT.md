## 1. エグゼクティブサマリー

### 1.1 ビジョン

**「AI開発を発注するための仕様書・相場・ベンダーマップを標準化して握るOS」を構築する。**

本プラットフォームは、単なるAI開発会社と発注企業のマッチングサービスではない。AI/DX導入における「情報非対称性」を解消し、**発注側の意思決定プロセスそのものを構造化・自動化するインフラストラクチャ**である。

### 1.2 解決する本質的課題

日本の中小企業がDX/AX（AI Transformation）を推進できない根本原因は、単に「開発費が高い」ことではない。真の課題は**「何を、どこに、いくらで頼めばいいか分からない」という情報非対称性**である。

この課題を解決するために、以下の3つの標準化を実現する：
- **仕様の標準化**：AI要件定義の言語化・テンプレート化
- **相場の標準化**：業界×ユースケース×規模別の価格レンジの可視化
- **ベンダーの標準化**：開発会社の実力・相性・信頼性の定量評価

### 1.3 競合優位性の源泉

既存のマッチングプラットフォーム（CrowdWorks、Lancers、AI Market等）は「案件と人材/企業を繋ぐ」という表層的な機能に留まっている。本プラットフォームは、**仕様とデータを握るレイヤー**に参入することで、以下の持続的競合優位性を構築する：

1. **データネットワーク効果**：案件が流れるほど相場精度が上がり、より多くのユーザーが集まる
2. **学習曲線効果**：要件定義AIが進化し続け、後発が追いつけない
3. **スイッチングコスト**：仕様書フォーマットが業界標準になれば、離脱困難

## 2. 解決すべき社会課題：日本のDX/AX領域における6つの「負」

### 2.1 負①：開発費が高く、投資リスクが致命的

**現状認識**
- 中小企業にとって、数百万〜数千万円のDX/AI投資は「一発外したら致命傷」レベルのリスク
- 中小企業白書でも「費用の負担が大きい」がDX推進の最大課題として顕在化
- 特に年商10億円以下の企業では、失敗時の再チャレンジ余力がほぼない

**構造的要因**
この「高い」という認識は、単純なコスト問題ではなく、**価格の妥当性を判断する基準がない**ことに起因する。同じ「チャットボット導入」でも、50万円から500万円まで見積もりが出る。発注者はどちらが適正か判断できず、結果として「高い」と感じるか、あるいは安すぎる業者に依頼して失敗する。

**本プラットフォームでの解決アプローチ**
- 相場の可視化により、見積もりの妥当性を即座に判断可能にする
- 「この要件なら、この価格帯が標準」という基準を市場に提供する

### 2.2 負②：DX/AI人材・リテラシーの絶対的欠如

**現状認識**
- 2030年には60〜80万人のIT人材不足が見込まれる（経済産業省推計）
- 特にAI・データサイエンス領域は深刻で、中小企業が採用することはほぼ不可能
- 「社内にエンジニアがいない」が前提の企業が日本企業の大多数

**構造的要因**
人材がいないことは、単に「作れない」だけでなく、**「何を作るべきか企画できない」「ベンダーの提案を評価できない」**という二次的問題を引き起こす。つまり、発注プロセス全体がブラックボックス化する。

**本プラットフォームでの解決アプローチ**
- AI要件定義ツール（アラジン方式）により、非エンジニアでも要件を言語化可能にする
- ベンダー評価の定量化により、専門知識なしでも選定判断を支援する

### 2.3 負③：「何をやるべきか分からない」企画力の欠如 & PoC地獄

**現状認識**
- 経営層自身がDXを「何かデジタルなこと」程度の理解で止まっている
- 「とりあえずチャットボット」「とりあえずRPA」で始めて、成果が出ずに終わる
- PoC（概念実証）を繰り返すだけで、本番運用まで至らない「PoC地獄」が蔓延

**構造的要因**
この問題の本質は、**ユースケースと業務課題のマッピングができていない**ことにある。「AIで何かしたい」という漠然とした願望と、「この業務のこの部分をこの技術で効率化する」という具体的なスコープの間に、巨大なギャップが存在する。

**本プラットフォームでの解決アプローチ**
- 業界×業務×技術の典型パターンをデータベース化
- 対話型ヒアリングで、漠然とした課題から具体的ユースケースを特定
- 「あなたの課題に近い導入事例」を提示し、成功イメージを具体化

### 2.4 負④：ベンダー情報の非対称性 & 選定コストの高さ

**現状認識**
- 「どこに頼めばいいか分からない」という根本的な情報欠如
- 各社のWebサイトを見ても、実力・得意分野・価格帯が比較できない
- 紹介経由で頼むしかなく、ミスマッチリスクが高い

**構造的要因**
AI開発業界は、**情報開示のインセンティブ設計が壊れている**。ベンダー側は「うちは何でもできます」と言い、具体的な得意分野や価格帯を明示しない。これは、特化すると案件を逃すリスクがあるためだが、結果として発注者は正確な情報を得られない。

**本プラットフォームでの解決アプローチ**
- ベンダーの強み・実績・価格帯を構造化されたフォーマットで可視化
- プラットフォーム経由の案件データから、客観的な評価指標を算出
- 「この要件なら、この3社が最有力」という具体的なレコメンドを提供

### 2.5 負⑤：レガシーシステム & 紙文化によるDX/AXの詰まり

**現状認識**
- 古い基幹システムがブラックボックス化し、誰も仕様を理解していない
- 紙・FAX・ハンコが現役で、データが電子化されていない
- 「AIを入れたい」以前に、データの断絶という根本問題がある

**構造的要因**
この問題は、**AI導入の前提条件の欠如**を意味する。多くのAI開発プロジェクトが失敗する原因は、AIそのものではなく、既存システムとの統合や業務フローとの整合性にある。ここを無視して「AIを導入しましょう」と言っても、砂上の楼閣になる。

**本プラットフォームでの解決アプローチ**
- 要件定義プロセスに「既存システム・データ状況」の深掘りを組み込む
- 「まずデータ整備が必要」という判定を正直に行い、フェーズ分けを提案
- レガシー対応に強いベンダーをタグ付け・可視化

### 2.6 負⑥：DX疲れ（過去の失敗による学習性無力感）

**現状認識**
- 一度DX/AI導入で失敗した企業は「もう騙されたくない」と疑心暗鬼
- 「またコンサル費だけ持っていかれるのでは」という不信感
- 結果として、本当に必要な投資も躊躇してしまう

**構造的要因**
失敗体験は、単なる金銭的損失ではなく、**組織の心理的な学習性無力感**を生む。「どうせ何をやっても上手くいかない」という認知が形成されると、合理的な意思決定すらできなくなる。

**本プラットフォームでの解決アプローチ**
- 透明性の高い相場・評価情報により、「騙されている」不安を解消
- 小規模・低リスクな導入から始めるステップ設計を提案
- 成功事例だけでなく「失敗パターンとその回避方法」も開示

## 3. プラットフォームコンセプト

### 3.1 ポジショニング

**「AI開発の発注OS」**

本プラットフォームは、以下の3層からなるポジショニングを取る：

```
┌────────────────────────────────────────────────────────────┐
│   【アプリケーション層】                                     │
│   マッチング・プロジェクト管理・決済                          │
├────────────────────────────────────────────────────────────┤
│   【OS層】★ 本プラットフォームのコア                         │
│   仕様書標準化 / 相場データベース / ベンダー評価エンジン       │
├────────────────────────────────────────────────────────────┤
│   【インフラ層】                                             │
│   AI開発会社 / フリーランス / ツール・クラウドサービス         │
└────────────────────────────────────────────────────────────┘
```

既存プレイヤー（CrowdWorks、Lancers、AI Market等）はアプリケーション層で競争している。本プラットフォームは**OS層を押さえる**ことで、上位レイヤーが依存せざるを得ないインフラになる。

### 3.2 コアコンセプト：「3つの標準化」

#### 3.2.1 仕様の標準化

**概念**
AI開発プロジェクトの要件定義を、どの企業に依頼しても通用する**標準フォーマット**で言語化する。

**具体的な標準化要素**
- ユースケース定義（何を自動化/効率化するのか）
- データ要件（必要なデータの種類・量・品質・アクセス方法）
- システム連携要件（既存システムとのAPI連携、データフロー）
- 非機能要件（セキュリティ、パフォーマンス、可用性）
- 成功指標（KPI定義と測定方法）
- 制約条件（予算、期間、技術スタック指定）

**標準化の効果**
- ベンダー間での見積もり比較が可能になる
- 曖昧な要件による炎上リスクが低減する
- 過去案件との類似性判定が可能になり、相場推定精度が向上する

#### 3.2.2 相場の標準化

**概念**
「業界 × ユースケース × 技術 × 規模」のマトリクスで、AI開発の**価格レンジを可視化**する。

**具体的なデータ構造**
```
相場レコード = {
  業界: "製造業",
  ユースケース: "外観検査自動化",
  技術パターン: "画像認識AI",
  規模: "単一ライン・月間1万件処理",
  価格レンジ: {
    下限: 300万円,
    中央: 500万円,
    上限: 800万円
  },
  期間レンジ: {
    下限: 2ヶ月,
    中央: 3ヶ月,
    上限: 5ヶ月
  },
  サンプル数: 23件,
  成功率: 78%
}
```

**相場データの活用方法**
- 発注者：見積もりの妥当性を即座に判定
- ベンダー：競争力のある価格設定の参考
- プラットフォーム：マッチング精度の向上

#### 3.2.3 ベンダーの標準化

**概念**
AI開発会社・フリーランスの実力・相性・信頼性を**定量的に評価・比較可能**にする。

**評価軸の設計**

| 評価カテゴリ | 評価項目 | データソース |
|------------|---------|-------------|
| 専門性 | 得意業界、技術スタック、実績数 | 自己申告＋案件実績 |
| 信頼性 | 納期遵守率、品質スコア、リピート率 | プラットフォーム計測 |
| 対応力 | 初回レス時間、見積もり提示日数、コミュニケーション評価 | プラットフォーム計測 |
| 適正価格 | 相場との乖離、価格帯 | 自己申告＋成約実績 |
| 稼働状況 | 現在の受注可能時期 | 自己申告 |

**可視化の粒度**
- 業界特化度：「製造業案件が80%」など
- 技術プロファイル：「RAG: ★★★★★、画像認識: ★★★☆☆」など
- 規模適性：「300万円以下の小規模案件に強み」など

## 4. ターゲット市場

### 4.1 プライマリターゲット

**日本の中小〜中堅企業（年商3億円〜300億円）**

**ターゲットの特徴**
- AIを業務に導入したいが、社内にエンジニア/データサイエンティストがいない
- IT投資の意思決定者は、情シス・経営企画・現場マネージャー
- 「とりあえず相談したい」「何ができるか知りたい」というリテラシーレベル
- 過去にDX/IT投資で失敗経験がある、または周囲の失敗を見ている

**ターゲット規模の推計**
- 中小企業（従業員300人以下）：約358万社
- うち、AI/DX投資に関心がある層：推計15〜20%（約50〜70万社）
- うち、実際に投資余力がある層：推計30%（約15〜20万社）

### 4.2 セカンダリターゲット

**大企業の事業部門・子会社**

**ターゲットの特徴**
- 本社IT部門を通さず、事業部独自でAI導入を検討
- 承認プロセスの関係で、相場感・ベンダー比較情報が必要
- 「情シスに任せると時間がかかる」「自分たちで調べたい」

### 4.3 サプライサイドターゲット

**AI開発会社・フリーランス（10〜100人規模）**

**登録を促すインセンティブ**
- 案件獲得チャネルの多様化
- 仕様が明確な案件が来るため、工数見積もりリスクが低減
- 相場情報へのアクセス（自社の価格競争力を把握）

## 5. コア機能と技術的アプローチ

### 5.1 機能①：AI要件ヒアリング＆仕様書自動生成（アラジン方式）

#### 5.1.1 コンセプト

ユーザーが「経理の請求処理を自動化したい」「コールセンターでAI使いたい」といった自然言語で入力すると、AIが対話的に深掘りを行い、**標準化された仕様書を自動生成**する。

このプロセスは、ゲーム「Akinator（アキネイター）」が「この人ですか？」とYes/No質問を繰り返して人物を特定するのと同様の原理を用いる。**ビット削減**の考え方を適用し、最小限の質問数で最大限の情報を取得する。

#### 5.1.2 技術的アプローチ

**ユースケースツリーの事前設計**

いきなりLLMに丸投げするのではなく、人力で設計した「ユースケースツリー」をベースにする。

```
業務領域
├── 経理・財務
│   ├── 請求書処理自動化
│   ├── 経費精算自動化
│   └── 予実管理高度化
├── 営業・マーケティング
│   ├── リードスコアリング
│   ├── 顧客問い合わせ対応
│   └── 提案書自動生成
├── カスタマーサポート
│   ├── FAQチャットボット
│   ├── 問い合わせ分類・ルーティング
│   └── 応対品質分析
├── 製造・物流
│   ├── 外観検査自動化
│   ├── 需要予測
│   └── 設備故障予知
└── ...
```

**質問設計の原理**

各質問は、可能性空間を最大限に二分割するように設計する。

```
Q1: 業種は？ → 可能性を1/10に絞り込み
Q2: 最も困っている業務領域は？ → 可能性を1/5に絞り込み
Q3: 現在その業務にかかる時間/コストは？ → 規模感を特定
Q4: 既存データの状態は？ → 技術的実現可能性を判定
Q5: 既存システムとの連携は必要？ → アーキテクチャを決定
...
```

10〜20問で、典型的なユースケースのうち1〜3個まで絞り込み、それぞれについて仕様書ドラフトを生成する。

**回答オプションの設計**

単純なYes/Noに加え、以下のオプションを用意：
- 「はい」「いいえ」
- 「部分的にそう」（条件付きYes）
- 「分からない」（追加質問で深掘り）
- 「該当なし」（この分岐をスキップ）

#### 5.1.3 生成される仕様書の構成

```markdown
# AI開発 要件定義書

## 1. プロジェクト概要
- プロジェクト名：〇〇業務自動化
- 依頼企業：株式会社△△
- 業界：製造業（精密機械）
- 想定予算：300〜500万円
- 希望納期：3ヶ月以内

## 2. 現状課題
- 対象業務：外観検査
- 現状の処理方法：目視検査（熟練工2名、1日8時間）
- 課題：人員不足、判定のばらつき、記録の未電子化

## 3. ゴール定義
- 定量目標：検査時間50%削減、不良見逃し率1%以下
- 定性目標：熟練工依存の解消、データに基づく品質管理

## 4. データ要件
- 入力データ：製品画像（現在カメラ設置済み、日次約500枚）
- 教師データ：過去の検査記録（紙、約3年分あり）
- データ品質課題：紙記録の電子化が必要

## 5. システム要件
- 連携システム：生産管理システム（オンプレ、API未対応）
- インフラ制約：社内ネットワーク内での稼働必須
- セキュリティ要件：機密図面を含むためクラウド不可

## 6. 技術アーキテクチャ案
- 推奨技術：画像認識AI（CNN/Vision Transformer）
- 推奨構成：エッジデバイス＋オンプレサーバー
- 代替案：クラウドAPI利用（セキュリティ要件緩和が前提）

## 7. 概算見積もり
- 相場レンジ：300〜600万円（類似23案件の実績より）
- 期間レンジ：2〜4ヶ月

## 8. 推奨ベンダー候補
- A社：製造業特化、画像認識実績多数、500万円帯中心
- B社：中小向け低価格、300万円帯、納期2ヶ月可
- C社：エッジAI専門、オンプレ構築に強み
```

#### 5.1.4 この機能が解決する「負」

- **負②（人材欠如）**：非エンジニアでも、対話するだけで要件が言語化される
- **負③（企画力欠如）**：「何をすべきか」の選択肢がシステム側から提示される
- **負④（情報非対称）**：ベンダーに丸投げせず、発注者側に仕様の主導権が生まれる

### 5.2 機能②：相場レンジ提示エンジン

#### 5.2.1 コンセプト

過去の案件データを学習し、新規案件の仕様書から**価格レンジと期間レンジを推定**する。

#### 5.2.2 データソース

**プライマリデータ（信頼度：高）**
- プラットフォーム経由の成約案件（業界・ユースケース・規模・金額・工数・満足度）
- 提携ベンダーからの匿名化された案件実績

**セカンダリデータ（信頼度：中）**
- 公開されている導入事例（補助金事例、ベンダー事例集から抽出）
- ベンダーからの自己申告（価格帯・主戦場）

**使用しないデータ**
- スクレイピングで取得したクラウドソーシングの案件情報（規約違反リスク、データ品質問題）

#### 5.2.3 推定ロジック

```
入力：仕様書（構造化データ）

処理：
1. 類似案件検索（業界×ユースケース×技術×規模でベクトル検索）
2. 類似案件の価格・期間分布を取得
3. 規模調整係数を適用（データ量、連携複雑度、カスタマイズ度）
4. レンジ出力（下限10%ile、中央値、上限90%ile）

出力：
- 相場レンジ：300〜600万円（中央値450万円）
- 期間レンジ：2〜4ヶ月（中央値3ヶ月）
- 類似案件数：23件
- 信頼度：高（サンプル数20件以上）
```

#### 5.2.4 見積もり比較機能

ベンダーから見積もりが届いた際、相場との乖離を可視化：

```
受領見積もり：A社 680万円 / 4ヶ月

相場との比較：
├── 価格：相場上限（600万円）より13%高い
├── 期間：相場範囲内
└── 総合判定：「やや割高。交渉余地あり、または高品質仕様の可能性」

参考情報：
- A社の過去案件平均：相場比+8%（品質評価は相場+0.5ポイント）
- 類似規模案件のA社実績：成功率92%
```

## 6. 蓄積するアセット（M&A観点での価値）

### 6.1 M&Aを狙う上での逆算思考

本プラットフォームは、2〜3年後のM&Aを視野に入れた設計を行う。買い手（大手SIer、通信キャリア、クラウドベンダー、既存クラウドソーシング企業）が「数十億円払ってでも欲しい」と思うアセットを逆算し、それを蓄積する。

### 6.2 蓄積すべき4つのアセット

#### 6.2.1 AI導入パターン×相場データベース

**内容**
- 「業界 × ユースケース × 技術 × 規模」のマトリクスで整理されたAI導入パターン
- 各パターンにおける価格レンジ・期間レンジ・成功率
- 典型的なトラブルパターンとその回避策

**価値**
- 競合が一朝一夕では構築できない「経験知のデータベース」
- AI導入コンサルティングの自動化基盤
- 相場の業界標準を握ることで、価格決定力を持つ

**蓄積方法**
- プラットフォーム経由のすべての案件を構造化データとして記録
- 提携ベンダーからの案件情報提供（匿名化）
- 公開事例からの抽出（補助金事例等）

#### 6.2.2 標準化された仕様書テンプレート

**内容**
- どのAI開発会社に投げても通用する、標準化された要件定義フォーマット
- 業界×ユースケース別のテンプレート（30〜50種類を目標）
- テンプレート利用実績と、その結果（成功率・トラブル発生率）

**価値**
- AI開発の発注プロセスそのものを標準化
- このフォーマットが業界標準になれば、すべての案件がこのフォーマットを通る
- ベンダー側も「このフォーマットなら見積もりやすい」と認識

**蓄積方法**
- アラジン（対話型要件定義）の改善ループを回し続ける
- テンプレート利用後のフィードバックを収集・反映

#### 6.2.3 ベンダー評価データ

**内容**
- AI開発会社・フリーランスの実力・相性・信頼性の定量評価
- 業界×技術軸での「誰が強いか」マップ
- 価格帯と品質のトレードオフ情報

**価値**
- 客観的なベンダー評価は、現在どこにも存在しない
- 発注者にとっては「信頼できる情報源」
- ベンダーにとっては「評価を高めればより良い案件が来る」インセンティブ

**蓄積方法**
- プラットフォーム経由案件の実績データを自動計測
- 発注者からの評価・フィードバック収集
- ベンダー自己申告情報の蓄積

#### 6.2.4 要件定義AIワークフロー

**内容**
- アラジン方式の対話ロジック・質問ツリー・判定アルゴリズム
- 自然言語からの仕様書生成プロンプト・モデル
- 継続的な改善によって蓄積されたノウハウ

**価値**
- 「AI開発の要件定義を自動化する」という機能そのものが製品になる
- 他社（SIer、コンサル）への OEM/ライセンス 販売が可能
- M&A時、この技術資産が評価対象になる

**蓄積方法**
- 実際の利用ログからの継続的改善
- 要件定義→プロジェクト結果のフィードバックループ

### 6.3 買い手にとって価値がないもの

以下は、M&A時にほぼ評価されない：
- 「自社が安く開発できます」という売り文句
- スクレイピングで集めた雑多な価格一覧（法的リスク、データ品質問題）
- 手作業で調整しないと回らない"なんちゃってマッチング"
- 仕様が標準化されていない、属人的なオペレーション

## 7. 競合との差別化

### 7.1 競合マップ

| プレイヤー | ポジション | 強み | 弱み |
|-----------|-----------|------|------|
| CrowdWorks | 汎用クラウドソーシング | 利用企業100万社、ワーカー710万人 | AI特化ではない、仕様標準化なし |
| Lancers | 汎用クラウドソーシング | AI人材強化中、ブランド認知 | 仕様・相場の標準化まで踏み込まず |
| AI Market | AI開発会社マッチング | AI特化、会社比較 | 人力コンシェルジュ型、スケール困難 |
| AI導入.com | AI導入支援マッチング | 導入支援に特化 | 仕様テンプレ・相場エンジン未整備 |
| アイスラボ | AI人材マッチング | AI人材特化 | 開発会社ではなく人材特化 |

### 7.2 本プラットフォームの差別化ポイント

**競合との決定的な違い**

```
【既存プレイヤー】
「AI開発会社を探して紹介します」
      ↓
人力コンシェルジュ型。スケールすればするほど人手が必要

【本プラットフォーム】
「AI開発の仕様・相場・ベンダー情報を構造化して提供します」
      ↓
データとアルゴリズムで標準化。スケールするほどデータが増えて精度が上がる
```

### 7.3 持続的競合優位性の構造

#### 7.3.1 データネットワーク効果

```
案件が増える → 相場データが増える → 相場推定精度が上がる
     ↑                                      ↓
発注者が増える ← 「相場が分かる」価値が上がる ←
```

一度この循環が回り始めると、後発は追いつけない。

#### 7.3.2 学習曲線効果

```
要件定義AIの利用回数が増える
     ↓
質問ツリー・判定ロジックが改善される
     ↓
要件定義の精度が上がる
     ↓
プロジェクト成功率が上がる
     ↓
プラットフォームの評判が上がる
     ↓
利用者が増える
```

#### 7.3.3 スイッチングコスト

仕様書フォーマットが業界標準になると：
- 発注者：過去の仕様書と比較可能。他に移ると履歴が途切れる
- ベンダー：このフォーマットに慣れる。他に移ると再学習コスト
- 結果として、離脱しにくくなる

## 8. 自然と流行る構造設計

### 8.1 なぜ「自然と流行る」必要があるのか

広告費を大量投下してユーザーを獲得するモデルは、資本勝負になる。本プラットフォームは、**構造的に利用者が増えていく**仕組みを持つことで、資本力に依存しない成長を目指す。

### 8.2 流行の構造①：発注者側のインセンティブ設計

**無料で提供する価値**
- 要件定義AIの利用（仕様書生成）
- 相場レンジの閲覧
- ベンダー一覧の閲覧

**なぜ無料でも収益化できるか**
- ベンダーマッチング時の成約手数料
- ベンダー向けプレミアム掲載料
- 相場データ・レポートの販売（法人向け）

**発注者にとっての離脱困難性**
- 仕様書の履歴がプラットフォームに蓄積
- 相場との比較機能は、他では得られない
- ベンダー評価情報も、ここでしか見られない

### 8.3 流行の構造②：ベンダー側のインセンティブ設計

**登録のメリット**
- 明確な仕様で案件が来るため、見積もりリスクが低い
- 自社の強みが可視化され、適切な案件がマッチングされる
- 相場情報にアクセスでき、競争力のある価格設定ができる

**良いベンダーほど優遇される仕組み**
- 納期遵守率・品質スコアが高いベンダーは上位表示
- リピート率が高いベンダーはバッジ表示
- 逆に、トラブルが多いベンダーは露出低下

**この設計の効果**
- 質の高いベンダーが集まる
- 質の高いベンダーがいるから、発注者も安心
- 良い循環が生まれる

### 8.4 流行の構造③：コンテンツとしての拡散性

**発注者が自然と言及する場面**
- 「〇〇で相場を調べたら、500万円くらいだった」
- 「〇〇の仕様書生成を使ったら、要件がスッキリした」
- 「〇〇でベンダー比較したら、A社が良さそう」

**ベンダーが自然と言及する場面**
- 「〇〇で評価4.5取れました」（ポートフォリオに掲載）
- 「〇〇経由の案件は仕様が明確で助かる」
- 「〇〇のフォーマットに沿った提案書を作成」

### 8.5 流行の構造④：検索エンジン最適化（SEO）

**狙うキーワード**
- 「AI開発 相場」
- 「AI開発会社 比較」
- 「チャットボット開発 費用」
- 「RAG 導入費用」
- 「〇〇業界 AI導入事例」

**コンテンツ戦略**
- 相場データをベースにした記事コンテンツ
- 業界×ユースケース別の導入ガイド
- 「AI開発を発注する前に知っておくべきこと」系の教育コンテンツ

## 9. ビジネスモデル

### 9.1 収益構造

#### 9.1.1 成約手数料（メイン収益）

- 成約金額の5〜10%をベンダーから徴収
- または、発注者からの手数料（相場より安くマッチングできた場合のシェア）

#### 9.1.2 ベンダー向け課金

- **プレミアム掲載**：上位表示、詳細プロファイル掲載
- **リード課金**：問い合わせ1件ごとの課金
- **認定バッジ**：プラットフォーム認定マークの付与

#### 9.1.3 データ・レポート販売

- 業界別相場レポート（法人向け月額or買い切り）
- カスタム分析レポート（特定業界・ユースケースの深掘り）
- ベンダー向け競合分析レポート

#### 9.1.4 自社開発チームの活用（マーケットメイカー機能）

- ベンダーが対応できない案件、小規模すぎる案件を自社チームで受注
- 目的は売上ではなく、**データ収集とテンプレート改善**
- 「リファレンス実装チーム」として位置づけ

### 9.2 価格設計の原則

**「安さ」を売りにしない**

「1円開発」「市場相場の1/3」といったポジショニングは取らない。理由：
- 自社の利益率を圧迫する
- 「安い＝低品質」のイメージがつく
- M&A時に「安い開発屋」と見なされ、資産価値が評価されない

**「適正価格の可視化」を価値にする**

- 「相場が分かるから、ぼったくられない」
- 「品質と価格のトレードオフが見えるから、合理的に選べる」
- この価値を提供することで、価格競争に巻き込まれない

## 10. 実行ロードマップ

### 10.1 フェーズ0：データ蓄積の開始（0〜6ヶ月）

**目標**
- 30〜50案件分のデータを蓄積
- 最初の相場レンジとテンプレートを構築

**実施内容**
1. **案件データのフォーマット統一**
   - すべての案件で以下を記録：
     - 業界
     - ユースケース
     - データ状況
     - 工数（実際）
     - 金額
     - 期間
     - 成果（KPI / 満足度）
   - Notion/スプレッドシートで十分

2. **ヒアリング質問テンプレートの作成**
   - クライアントとの初回ヒアリング項目を固定化
   - 毎回同じ質問をすることで、データが比較可能になる

3. **仕様書ドラフト自動生成の試作**
   - 質問テンプレート＋回答 → LLMで仕様書ドラフト生成
   - 手動で修正しながら、プロンプトを改善

4. **ベンダーネットワークの構築開始**
   - 既存のネットワークで10〜20社を集める
   - 得意領域・実績・希望単価を管理

### 10.2 フェーズ1：簡易プロダクト化（6〜12ヶ月）

**目標**
- Web上で要件定義〜ベンダー紹介まで完結
- プラットフォーム経由の案件を10〜30件成約

**実施内容**
1. **Webプロダクトの構築**
   - ヒアリングチャット（AIとの対話フォーム）
   - 仕様書PDFの自動生成
   - ベンダー一覧表示（簡易版）

2. **相場レンジ機能（簡易版）**
   - フェーズ0で蓄積したデータから、手動でレンジを設定
   - 完全自動推定は次フェーズ

3. **マッチングは半手動**
   - 仕様書が生成されたら、担当者がベンダーを推薦
   - 完全自動マッチングはまだ早い

4. **データ蓄積の継続**
   - すべての案件でフォーマット化されたデータを記録
   - 成約後の満足度・トラブルも記録

### 10.3 フェーズ2：スケール化（12〜24ヶ月）

**目標**
- 相場エンジン・マッチングロジックの自動化
- 月間成約100件規模へ

**実施内容**

## 11. 成功のための原則

### 11.1 原則①：「安さ」から「標準化・相場・データ」へ軸を移す

- 「安いです」は他社もできるし、長期でブランドを壊す
- 「AI開発の仕様と相場が分かる場所」は、真似されにくいし、データが積み上がる
- 価格で勝負せず、情報で勝負する

### 11.2 原則②：案件ごとに「資産」を意識する

各案件から必ず抽出すべきこと：
- テンプレ化できる仕様項目は何か
- 共通するアーキテクチャパターンは何か
- 相場感にズレはあったか

これを毎回振り返らないと、2〜3年経っても「ちょっと経験豊富な受託屋」で終わる。

### 11.3 原則③：プラットフォームは「楽なビジネス」ではないと自覚する

両面市場（発注側・受注側）を立ち上げるのは、正直かなりキツい。だからこそ：
- 一番痛い「負」はどこか
- そこに対して何を「標準化」して握るのか

ここを徹底的に絞る必要がある。

### 11.4 原則④：「影響力」を「プラットフォーム」に変換する

HIKAKINがUUUMを作ったように、個人の影響力を他人が乗れるプラットフォームに変える。

本プラットフォームの場合：
- 「AIを使って爆速で検証・開発できるチーム」という影響力を
- 「AI導入の仕様と相場のOS」に変換する

個人の労働時間に依存するモデルから、データとアルゴリズムが働くモデルへ。

### 11.5 原則⑤：データ設計を先に行う

走り始める前に、各案件で絶対に取るべき項目をフォーマット化する。

ここが曖昧なまま走ると、2〜3年後に「惜しいけど汚いデータの山」が残る。

**最初から記録すべき項目**
```yaml
案件基本情報:
  案件ID: 一意識別子
  業界: enum
  ユースケース: enum
  技術パターン: enum
  規模: enum（小/中/大）

見積もり・実績:
  見積もり金額: number
  見積もり期間: number
  実際金額: number
  実際期間: number
  乖離率: 自動計算

ベンダー情報:
  担当ベンダー: reference
  初回レス時間: number
  見積もり提示日数: number

プロジェクト結果:
  成功/失敗: enum
  納期遵守: boolean
  品質スコア: 1-5
  発注者満足度: 1-5
  トラブル有無: boolean
  トラブル内容: text（あれば）

学習ポイント:
  テンプレ化可能項目: text
  相場からのズレ: text
  改善点: text
```

## 付録A：用語定義

| 用語 | 定義 |
|------|------|
| DX (Digital Transformation) | デジタル技術を活用したビジネスモデル・業務プロセスの変革 |
| AX (AI Transformation) | AI技術を活用したビジネスモデル・業務プロセスの変革 |
| PoC (Proof of Concept) | 概念実証。本格導入前の小規模検証 |
| RAG (Retrieval-Augmented Generation) | 外部データを検索・参照しながら回答を生成するAI手法 |
| LLM (Large Language Model) | 大規模言語モデル。GPT、Claude等 |
| アラジン方式 | Yes/No質問を繰り返して対象を特定する手法（Akinatorから着想） |
| ビット削減 | 情報理論に基づき、最小限の質問で最大の情報を得るアプローチ |
| 両面市場 | 発注者と受注者の双方がユーザーとなるプラットフォームモデル |
| ネットワーク効果 | ユーザーが増えるほどサービス価値が上がる現象 |
| スイッチングコスト | 他サービスに移行する際に発生するコスト（金銭・学習・履歴喪失等） |
